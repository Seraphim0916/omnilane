## Decision criterion

Treat omnilane as a public, local-first model-routing CLI for individuals and
small teams. It should route work safely across supported subscription CLIs,
fail clearly, preserve user state, remain usable on stock macOS and Linux, and
provide an optional loopback-only observer without exposing private worker
logs. Improvements must be incremental, independently reviewable, and backed
by automated checks plus the real CLI, API, browser, or provider surface they
change.

## Scope and constraints at audit time

- Audit all tracked code, tests, documentation, install/configuration paths,
  GitHub Actions, and the optional Live UI.
- Do not merge, push, publish, delete worktrees, or modify `main`.
- Provider behavior requiring paid calls is marked unverified unless exercised
  through an already-authorized local subscription CLI.
- Separate verified facts, inference, and unknowns.

## Audit stream A: adversarial correctness and security

The independent Codex Terra stream reached its watchdog before producing a
final report. Its intermediate activity is not treated as an independent
conclusion. Controller reproductions found:

- Job directories and prompt/result files inherit permissive ambient modes.
- `meta.json` can become invalid when legal path or override values contain
  JSON control characters.
- `jobs.sh` accepts path traversal in its job ID argument.
- The installer overwrites a pre-existing non-symlink global wrapper.

## Audit stream B: product and domain-standard gaps

The requested independent Opus audit could not run because the provider usage
limit was reached. This is an unavailable evidence stream, not a clean report.

Controller review verified these public-product gaps:

- A fresh `omnilane jobs list` prints nothing but exits `1` rather than
  succeeding with an empty list.
- Uninstall performs its cleanup but exits `1` when no provider CLI remains.
- There is no `doctor` command that checks routing syntax, watchdog support,
  provider binaries, writable state, or optional UI dependencies.
- Standard contribution, vulnerability-reporting, and durable changelog files
  are absent. This is lower priority than correctness and privacy fixes.

## Audit stream C: maintenance, portability, and cost

The Gemini 3.5 Flash stream produced a report before ending on provider
capacity. Its claims were treated as partial and independently checked.

Verified:

- Job storage has no retention mechanism. An isolated local sample confirmed
  that completed job state accumulates without a bound.
- Six of 27 Python tests are skipped in GitHub Actions because the browser
  harness only accepts the macOS Chrome path and CI does not install Python
  Playwright. The report claimed ten skipped tests; the authoritative CI log
  shows six.
- An empty lock directory with no PID is never reclaimed and reaches exit `87`
  after the full lock timeout.
- A missing executable in the first `exec` candidate is selected as available,
  then fails with exit `2`; the working second candidate is never tried.

Unverified or rejected:

- Grok safety refusals causing five paid retries is plausible but provider-text
  dependent and was not exercised.
- Generic model-name preflight is not possible from CLI presence alone. The
  verified narrower defect is the locally checkable `exec` candidate.
- Vote Round 2 failure does not discard Round 1 output; the result file remains
  available. That claim is rejected.
- The reported vote trap leak was not shown by a reproduction and existing
  code releases the Codex lock after each voter. It remains unproven.

## Investment decision and staged route

Proceed incrementally. The product has a useful core and passing release
checks, so no rewrite is justified. Correctness, privacy, and state-management
defects have direct user impact and small-to-medium fixes. Product polish and
governance remain behind runtime risks.

Recommended order:

1. Finish P0 safety fixes.
2. Harden jobs state and locking, then add conservative retention.
3. Restore browser coverage in CI.
4. Forward-port the whole-job deadline on top of v0.4.0.
5. Improve local diagnostics and project governance.

Each stage will define a stop signal. A candidate is rejected when it has no
reproducible user impact, duplicates an existing control, broadens security or
provider scope without authorization, or costs more complexity than the risk
it removes.

## Stop-condition audit

Three audit passes covered all tracked Shell/Python/UI code, the installer,
routing/configuration, job state, CI, docs, and existing worktrees. The final
pass found no remaining reproducible, high-value defect within the authorized
scope. Remaining observations were rejected or deferred deliberately:

- Provider-specific Grok refusal retries remain unverified without a stable
  provider response fixture; no speculative text classifier was added.
- Antigravity's permission flag and print-mode limitations are already
  documented and cannot be safely changed without current upstream behavior.
- Relative third-party symlink layouts, cosmetic UI changes, and automatic job
  deletion were not pursued because supported install paths already work or
  the risk exceeds the demonstrated benefit.

The original stale `codex/job-timeout-deadline` experiment was not used;
`codex/whole-job-deadline-v2` superseded it. Candidate worktrees were clean at
the close of this historical audit.
